## 0. 프로젝트 개요 — Orkis

자연어 질문을 입력하면 **LLM/RAG 서버가 SQL을 생성하고, 사용자가 등록한 DB에 실제로 실행해 결과를 반환**하는 Text-to-SQL 연구 플랫폼.

클라우드(웹) 버전과 데스크탑(단일 머신) 버전을 **하나의 코드베이스**로 동시 지원하며, 7개 저장소로 구성됨.

## 1. @orkis/core — 자체 백엔드 프레임워크 설계·구현 (285 커밋)

Node.js/TypeScript 환경에서 **Spring Boot 스타일의 데코레이터 기반 개발**을 가능하게 하는 프레임워크를 직접 설계.

- **DI 컨테이너**: `@Controller`/`@Service`/`@Component`/`@Dao`/`@Autowired` 데코레이터, ComponentScan 자동 빈 등록. 메타데이터 저장(데코레이터)과 인스턴스화·주입(BeanResolver)을 책임 분리.
- **선언적 트랜잭션(`@Transactional`)**: 메서드 래핑 기반 실행, **전파 정책(REQUIRED / REQUIRES_NEW / SUPPORTS)** 지원. 트랜잭션 컨텍스트를 Bean clone + Lazy Proxy로 격리·전파.
- **멀티 DB 추상화**: `BaseAdapter` 하나로 PostgreSQL·SQLite·MariaDB·Redis·File을 통일. `supportsTransaction()`로 어댑터별 트랜잭션 지원 분기.
- **동적 DB 연결(멀티테넌시)**: 요청의 userId로 사용자별 DB 정보를 조회해 런타임에 connection을 동적 생성 — Text-to-SQL의 핵심 인프라.
- **SSE**: `@SSEConnection` 데코레이터 + `SSEStream` 유틸 (멀티라인 직렬화·flush·onClose).
- **HTTP 클라이언트**: undici 기반 keep-alive 연결 풀링 (Agent / Pool dispatcher).
- **인터셉터·운영 기능**: path 패턴 매칭 인터셉터, 파일 업로드, Unix Domain Socket, Redis 세션 플러그인, CORS·로깅·글로벌 에러 핸들러, k8s 환경 대응.

## 2. orkis-backend — 백엔드 애플리케이션 (327 커밋)

- **OAuth 소셜 로그인**: 네이버·카카오·구글을 단일 인터페이스(`OAuthConfig` 맵 + `processLogin`)로 통합. authorization code → token → userInfo 정규화 → 신규/기존 분기. bcrypt 패스워드 로그인 병행, Bearer 토큰 인증.
- **동적 DB 연결 적용**: 코어의 `prepareDynamicDBConnection`을 백엔드에 적용, 등록된 연결정보 복호화 → 동적 어댑터 주입 → 사용자 DB liveness 측정(`DbConnectionChecker`).
- **RAG/LLM 챗 SSE 스트리밍**: AI(RAG)가 Redis 스트림에 쓰면 폴링 → `message_stream` 이벤트 변환 → 프론트 SSE. 토큰 스트리밍·step stepper·제목 생성·SQL 실행·취소 전 구간 처리.
- **세션 Redis 전환**: 세션 스토어를 Redis로 분리(`SessionPlugin`), 멀티 Pod 세션 공유.
- **단일 코드베이스 cloud/desktop 빌드 아키텍처**: TS `paths` fallback(`@/* → [desktop, main]`) + `tsc-alias`로 같은 import가 빌드 타깃에 따라 desktop override 우선 / main fallback으로 해석되게 설계.
- **인프라/배포**: Dockerfile 멀티스테이지, k8s 무중단 롤링 업데이트(`maxSurge:1/maxUnavailable:0`), PM2 클러스터 + PG Pool 검증, Jenkins CI.

## 3. orkis-jobs — 백그라운드 작업 서비스 (58 커밋, 단독에 가까움)

- **회복탄력 Redis 인프라**: ioredis 기반 `RedisConnection`/`RedisManager` — exponential backoff + jitter 재시도(최대 10회), 6종 이벤트 리스너, ping 헬스체크, 다중 커넥션 동시 초기화/graceful disconnect.
- **세션 라이프사이클 Job**: `ChatSessionCleanupJob`(타임아웃 강제 완료), `RedisCleanupJob`(완료 세션 5초 간격 삭제), `DataArchiveJob`(세션 데이터를 NAS에 `{chatroom_id}/YYYY-MM-DD.json` 아카이빙).
- **스케줄링 + 외부 트리거 통합**: node-cron `JobScheduler`로 cron·지속·수동 트리거 일원화. 아카이빙을 cron pull → **HTTP push 트리거**(`POST /archive/internal` + 큐잉)로 전환.
- **DI 마이그레이션**: 수동 `new`/정적 `DatabaseManager`를 1000줄 이상 제거하고 `@Service`/`@Autowired`/`@InjectConnection` 기반으로 전환, 도메인별 폴더 재편.
- **K8s/CI**: Jenkinsfile, deployment/ClusterIP svc, ConfigMap/Secret 주입, NodePool 라벨 정책.

## 4. orkis-desktop — Electron 데스크탑 앱 (82 커밋, 단독 소유)

웹 Orkis를 Electron 39 + electron-vite 데스크탑 앱으로 패키징하고, backend/AI를 **로컬 Podman 컨테이너로 구동하는 런처**.

- **OS 네이티브 통합**: OAuth 인증 창, 플랫폼별 타이틀바(mac hiddenInset), system tray, dmg/pkg·nsis·AppImage 패키징.
- **Podman 컨테이너 런처**: backend/AI를 PyInstaller 바이너리 → Podman 컨테이너 구동으로 전환(`container.service.ts`).
- **Windows 크로스플랫폼**: mac / Windows-NAT / Windows-mirrored 3개 네트워크 모드를 `ContainerNetConfig` 단일 함수로 집중(.wslconfig 자동 감지, WSL 어댑터 IP 추출).
- **Health 모니터링**: `PodmanEventsMonitor` — `podman events --format json`을 파싱해 `service:ready/unhealthy/down`을 IPC로 push.
- **부트스트랩 재설계**: `OrkisApplication` 클래스로 라이프사이클 일원화, config/container/ipc/health/services/errors 폴더 응집.

## 8. k8s-config — 연구소 쿠버네티스 플랫폼 직접 구성 (GitOps)

연구소 인프라 전체를 **매니페스트로 코드화(Infrastructure as Code)**한 저장소. KT Cloud ManagedKS(매니지드 쿠버네티스) 위에 클러스터·네트워킹·스토리지·CI/CD·데이터 서비스를 직접 설계·구축. *(`base/` `infrastructure/` `cicd/` `applications/` 구조로 도메인 분리)*

- **클러스터 구성 / 노드 역할 분리**: 7노드·28파드 규모 클러스터를 `nodeSelector` 라벨(`app=ingress` / `app=jenkins` / `app=orkis-database` / `app=orkis`)로 워크로드별 노드 풀 분리. Bitnami metrics-server(Helm)로 리소스 모니터링(OpenLens), `system-cluster-critical` 우선순위·노드 장애 시 30초 빠른 복구 toleration 적용.

- **인그레스 / L4 LB / 무중단 진입점**: ingress-nginx를 **DaemonSet + hostNetwork**로 인그레스 전용 노드에서만 80/443 listen, KT Cloud L4 LB가 해당 노드로 패스스루. IngressClass를 `nginx-k8s` → `nginx`로 마이그레이션.

- **TLS 자동화**: cert-manager + Let's Encrypt(`ClusterIssuer` prod/staging, HTTP-01 챌린지)로 `orkis.kr`·`git.orkis.kr` 인증서 자동 발급·갱신. staging 선검증 후 prod 전환으로 rate limit 회피.

- **SSE 전용 인그레스 설계**: `/api/sse/* → /sse/*` 경로 rewrite, **쿠키 기반 Sticky Session**(affinity), `proxy-buffering off`, read/send timeout 1800s로 장시간 스트리밍 연결 유지. (프론트 라우팅 인그레스와 분리)

- **스토리지(NFS 동적 프로비저닝)**: KT Cloud NAS(`172.25.1.129`)를 nfs-subdir-external-provisioner로 동적 PVC화. StorageClass 3종 설계 — `nfs-retain-prod`(Retain·archiveOnDelete), `nfs-delete-dev`(Delete), `nfs-static`(정적 매핑). 공유 PVC로 sqlitedb·chat-summary·orkis-jobs 데이터 영속화.

- **사내 CI/CD 플랫폼 직접 호스팅**:
  - **GitLab을 k8s 위에 자체 호스팅**(deployment·ingress·PVC, `git.orkis.kr`).
  - **Jenkins on k8s + Kubernetes Plugin**: 빌드 시 Agent Pod를 동적 생성→종료 시 자동 삭제(ephemeral). 최소 권한 RBAC 설계(`jenkins-deployer` SA를 orkis 네임스페이스에 바인딩, cicd↔orkis 크로스 네임스페이스 RoleBinding).

- **백업 / 재해 복구 자동화(CronJob)**:
  - **GitLab 소스 미러 백업**(매일 03:00 KST): GitLab API 페이지네이션으로 전 프로젝트 증분 `git fetch`, 일요일 tar.gz 스냅샷 + 30일 보존. **토큰을 디스크 origin URL에 저장하지 않아**(oauth2 URL only) NAS 백업 시 노출 최소화.
  - **GitLab 전체 백업**(분기 1회): `gitlab-backup create`, 90일 초과 gzip·365일 초과 삭제.

- **데이터 서비스 운영**: PostgreSQL(deployment·secret·configmap·로컬 PV+StorageClass), **Redis 3종 분리**(chat / stage / session, `redis:7-alpine`)를 DB 전용 노드(`app=orkis-database`)에 배치.

- **애플리케이션 배포(orkis 네임스페이스)**: front·backend·jobs·ai-service를 인그레스·PVC·ConfigMap/Secret로 묶어 배포. `orkis.kr` 진입은 프론트로, `/api/*`는 프론트 nginx가 백엔드 프록시, `/api/sse`는 인그레스가 백엔드로 직접 라우팅.
