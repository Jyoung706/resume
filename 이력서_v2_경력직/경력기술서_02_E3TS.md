## 1. 그룹웨어 채팅 — 서버·클라이언트 풀스택 (단독)

### 개요 · 규모

병원 그룹웨어의 실시간 메신저를 **설계부터 배포까지 서버·클라이언트 모두 단독**으로 개발. 회사(테넌트)별로 격리된 다중 조직이 한 클러스터 위에서 동작해야 하는 멀티테넌시 요구가 핵심 제약이었다.

| 구분 | 규모 |
|------|------|
| 채팅 서버(`cluster-chat-server`) | 본인 309커밋 / REST 48 엔드포인트 / **Socket 이벤트 8종** / JS 84파일 |
| 채팅 클라이언트(`cluster-chat-client`, 제품명 UEZTalk) | 본인 406커밋 / **Vue 컴포넌트 86** / Pinia store 10 / JS 33파일 |

- Socket 이벤트 8종(`src/socket/routers/socket.routes.js`) : `signIn` / `sendChat` / `joinRoom` / `leaveRoom` / `leaveAllRooms` / `readChat` / `exitRoom` / `inviteRoom`

### 역할

요구 분석·아키텍처 설계·서버 구현·Vue/Electron 클라이언트 구현·온프레미스 배포·고객사 현황 보고까지 채팅 도메인 전 과정을 단독 수행.

### 핵심 구현

**서버 (`cluster-chat-server`)**

- **PM2 cluster + Redis 어댑터 부하분산** — `ecosystem.config.cjs`에서 `exec_mode: 'cluster'`, `instances: Math.ceil(os.cpus().length / 2)`(CPU 코어 절반), `max_memory_restart: '1G'`, `autorestart: true`, `restart_delay: 4000`로 다중 워커를 자동 운영. 워커 간 메시지 브로드캐스트는 `src/data/connection/redis.connection.js`에서 `IoRedis.Cluster`로 pub/sub 클라이언트를 만들어 `io.adapter(createAdapter(pubClient, subClient))`(`@socket.io/redis-adapter`)로 통합 → 어느 워커에 붙은 소켓이든 전체에 전파.
- **회사별 namespace 멀티테넌시** — `io.of('/' + companyId)`로 회사마다 독립 통신 채널을 격리(`src/socket/socket.js`, `io.server.js`의 `setupNamespace()`). `userId`/`companyId` 이중 인증으로 테넌트 간 메시지 누수를 차단.
- **MongoDB + MySQL 하이브리드** — 메시지/대화는 MongoDB(Mongoose), 사용자/회사 메타는 MySQL. `src/data/connection/mongo.connection.js`에서 namespace별로 `mongoose.createConnection(dbUri)`을 만들어 `dbConnections{}`에 보관, `getDBConnection(namespace)`로 회사별 독립 Mongo 연결을 제공.
- **분산 락** — `src/utils/distributeLock.js`의 `DistributedLock` 클래스가 Redis `SET key value NX PX ttl`로 락을 **원자적으로 획득 + 자동 만료(데드락 방지)**, 해제는 Lua 스크립트로 소유자(lockValue) 검증 후 삭제. 클러스터 환경에서 단일 인스턴스만 스케줄러를 실행하도록 보장.
- **JWT 만료 자동 갱신** — 소켓 핸드셰이크(`socket.js`)·REST 미들웨어(`auth.middleware.js`) 양쪽에서 `jwt.TokenExpiredError`를 catch → `refreshTokenCheck()`로 재발급(`res.cookie('jwt', ...)`)해 연결 끊김 없이 세션 유지.

**클라이언트 (`cluster-chat-client`)**

- **Pinia `socketStore` 다중 namespace 동시 연결** — `src/stores/socketStore.js`의 `initAllNameSpace()`가 회사별 namespace 소켓을 `Promise.all`로 **병렬 초기화**, `connections`(reactive)에 type별로 보관하고 `getSocket` computed로 활성 namespace를 선택. 수신 이벤트 9종(`newChat`, `createChat`, `exitRoom`, `receivedChat`, `readChat`, `invitedRoom`, `newDM`, `readDM`, `reload:build`)을 `eventCallbacks` 레지스트리에 `onEvent`/`offEvent`로 동적 등록·해제.
- **Electron 크로스플랫폼 데스크탑 앱** — `electron/main.js`에서 `contextIsolation: true`/`nodeIntegration: false`로 보안 격리, IPC로 알림·다운로드(`net.request` 스트리밍 + 진행률)·파일 열기를 처리, 시스템 트레이 상주, `env.json` 동적 로딩, 단일 인스턴스 락. `electron-builder`로 mac(dmg)·Windows(NSIS) 빌드, `build/installer.nsh`에서 Windows 시작프로그램 레지스트리(`HKCU\...\Run`) 등록·제거를 커스터마이즈.
- **배포 자동 갱신** — 서버 배포 시 소켓 `reload:build` 이벤트를 받아 접속 중인 클라이언트를 자동 새로고침(`AppLayout.vue`의 `reloadWebhookHandler`). `.gitlab-ci.yml`로 특정 브랜치 push 시 Wine 기반 러너에서 Windows 빌드→배포 경로 복사.

### 트러블슈팅 (STAR)

**T1. PM2 cluster에서 cron이 워커마다 중복 실행 — 분산 락으로 단일 실행 보장**

- **상황(S)** : 부하 분산을 위해 PM2 cluster로 CPU 절반만큼 워커를 띄우자, `node-cron` 예약 작업(예: 예약 DM 발송)이 **모든 워커에서 각각 실행**돼 동일 메시지가 중복 전송될 수 있는 구조가 됐다.
- **과제(T)** : 워커를 늘려도 스케줄러는 **클러스터 전체에서 한 번만** 실행되도록 보장해야 했다.
- **행동(A)** : 인스턴스 번호로 분기하는 방식(`server.js`의 `WAS_SERVER_ID===1 && INSTANCE_ID===0`) 대신, Redis `SET NX PX` 기반 `DistributedLock`을 도입하고 `cron_functions.js`의 `runCronWithLock`으로 모든 인스턴스가 동일 락을 경합하게 만들어, 락을 획득한 단일 인스턴스만 작업을 수행하도록 정착시켰다(`app.js`에서 import). TTL(`PX`)로 인스턴스가 죽어도 락이 자동 만료돼 데드락을 방지.
- **결과(R)** : 워커 수와 무관하게 예약 작업이 정확히 1회 실행. 인스턴스 토폴로지에 의존하지 않아 스케일 변경에도 안전한 구조 확보.
- **근거 커밋** : `71db9d9 [Fix] util distribute lock & cron function fix runCronWithLock`, `7e6b6b0`·`86bf1bf`·`9fdd7d8`·`0f92ed7 [Fix] redis lock test`(락 튜닝 반복).

**T2. 분산 락이 PM2 프로세스 ID로는 인스턴스를 구분하지 못함 — NODE_APP_INSTANCE로 교정**

- **상황(S)** : 분산 락/로깅에서 워커를 식별하려 일반 프로세스 ID(`p_id`)를 쓰자, PM2 cluster 환경에서 인스턴스 구분이 의도대로 되지 않았다.
- **과제(T)** : PM2가 각 워커에 부여하는 **정식 인스턴스 식별자**로 교체해 락 소유/로그 추적의 정확도를 높여야 했다.
- **행동(A)** : 식별 기준을 PM2가 주입하는 `process.env.NODE_APP_INSTANCE`로 통일하고, `ecosystem.config.cjs`의 `instance_var: 'INSTANCE_ID'`와 정합을 맞췄다.
- **결과(R)** : 워커별 식별이 안정화되어 락 동작·로그 추적이 일관됨.
- **근거 커밋** : `60f7373 [Fix] distributeLock.js fix p_id => NODE_APP_INSTANCE`, `0ac03ce [Fix] server.js instance id check`, `7903dc9 [Fix] test pm2 instanceId check test`.

## 2. 병원 발렛파킹 주차관제 시스템

### 개요 · 규모

병원 발렛파킹 운영을 위한 주차관제 백엔드(`parking-server`)를 담당. 입·출차 상태를 관리자·요원 화면에 **실시간 반영**하고, HTTP와 WebSocket을 동일한 비즈니스 로직으로 처리하는 것이 핵심.

- 본인 385커밋 / **REST 119 엔드포인트** / **Layered 3계층**(Controller 21 · Service 21 · DAO 22) / JS 83파일
- 라우터 6개(admin/agent/exit/management/sms/user) · HTTP 메서드 분포 GET 45 / PATCH 35 / POST 26 / DELETE 13

### 역할

3계층 아키텍처 설계와 전체 API 구현, 실시간 소켓 통신, 인증·세션 제어, 다중 DB·푸시 연동을 단독 수행.

### 핵심 구현

- **Layered 3계층 아키텍처** — `controllers / services / dao`를 도메인(admin·agent·exitStep·management·sms·user)별로 미러링해 119개 REST API를 일관된 구조로 구현.
- **HTTP·WebSocket 동일 비즈니스 로직 통합** — `server.js`에서 `http.createServer(app)` 단일 HTTP 서버 위에 `initSocket(server)`로 Express와 Socket.io가 같은 서버를 공유. 입차/출차 REST 처리(`agent.entry.controller.js`, `exit_step.controller.js`)가 DB 트랜잭션 직후 동일 흐름에서 `codeEmit({op:"104000"...})` 소켓 브로드캐스트 + `sendWebPush(result)`까지 수행해, 한 번의 처리로 화면 실시간 갱신과 푸시를 함께 발생.
- **소켓 JWT 쿠키 인증 + 중복 로그인 강제 종료** — `utils/socket.js`에서 `socket.handshake.headers.cookie`를 파싱해 `tk` 토큰을 `verifyJwt`로 검증, 사용자별 `connectedUsers` 맵으로 세션을 추적. 동일 사용자의 새 접속을 감지하면 기존 연결에 `duplicateLogin`을 emit하고 `previousSocket.disconnect(true)`로 강제 종료.
- **TypeORM 다중 데이터소스** — `utils/databaseConnect.js`에서 메인 DB(`dataSource`, MySQL)와 SMS DB(`smsDataSource`, MariaDB)를 각각 `.initialize()`로 독립 연결.
- **Web Push 알림** — `utils/webpush`에서 VAPID(`setVapidDetails`) 설정 후, 주차 상태 변경 시 구독자 전체에 `webpush.sendNotification` 푸시.

### 트러블슈팅 (STAR)

**T5. 중복 로그인 시 이전 세션이 살아남는 문제 — connectedUsers 맵 기반 강제 종료**

- **상황(S)** : 한 계정으로 여러 곳에서 접속하면 이전 소켓 세션이 남아 상태가 꼬이고 알림이 중복되는 문제가 있었다.
- **과제(T)** : 동일 사용자의 신규 로그인 시 **기존 세션을 확실히 끊고** 최신 1세션만 유지해야 했다.
- **행동(A)** : `connectedUsers` 맵에 사용자(uIDX)→소켓을 보관하고, 신규 연결/로그인 시 기존 소켓에 `duplicateLogin` emit 후 `disconnect(true)`로 강제 종료. REST 로그인 컨트롤러(`user.sign.controller.js`)에서도 `disconnectUser(uIDX)` + `deleteConnectedUsers(uIDX)`를 호출해 **HTTP 인증 흐름이 WebSocket 세션을 직접 제어**하도록 통합.
- **결과(R)** : 중복 로그인 상태 정합성 확보, 유령 세션·중복 알림 제거.

**T6. Socket.io polling 폴백으로 인한 불안정 — WebSocket transport 고정**

## 3. 사내 인프라 / 형상관리 + 그룹웨어 메일

### 개요

소수 인원이 ssh-remote로 한 폴더를 공유하던 개발 환경의 한계를 직접 진단하고, **형상관리·CI/CD·메일까지 사내 인프라 전반을 단독 구축**. 온프레미스 인프라 역량의 출발점이 된 작업.

### 형상관리 / CI·CD (주도)

- **상황** : 개발 인력이 늘며 ssh-remote 공유 작업에서 코드 충돌이 잦아짐.
- **행동** : 대표·팀장에게 사내 Git 도입을 먼저 건의하고, **GitLab CE·Jenkins를 온프레미스에 단독 구축**(설치~초기 세팅). GitLab Webhook + Jenkins로 CI/CD 파이프라인을 구성하고, 서비스 배포용 Nginx 웹서버·도메인 SSL·DNS 레코드를 직접 구성. Git을 모르는 팀원에게 컨벤션과 Git flow를 직접 교육해 정착을 주도.
- **결과** : ssh-remote 공유 작업을 **레포 기반 로컬 개발 + 무중단 배포 환경**으로 전환. 이 경험이 이후 범익 연구소의 KT Cloud Kubernetes GitOps 구축으로 확장됨.

### 그룹웨어 메일

- **구축** : 오픈소스 기반 웹메일(Postfix·Dovecot·Roundcube)을 직접 구축하고, 상용 **Emailengine**으로 그룹웨어와 Gmail/Naver/Daum/웹메일 릴레이를 연동. Express.js로 메일엔진 호환 서버 API·릴레이↔그룹웨어 자동화 API·유저 등록 API를 구현하고 Vue.js 프론트엔드까지 담당.
- **문제 해결(STAR)** : 네트워크·메일 프로토콜 지식이 전무한 상태에서 SSL 연동 에러가 검색으로 풀리지 않자, 1~2주간 로그와 수·발신 서버 config의 주석까지 정독해 설정 한 줄을 찾아 해결. Gmail 발신 문제는 **SPF/DKIM/DMARC**로, 보안은 **WAF(ModSecurity)**로 대응.
- **결과** : 외부 메일 서비스와 호환되는 그룹웨어 메일 송수신 체계를 확보하고, 모르는 영역도 원인까지 파고들어 동작시키는 강점을 입증.
