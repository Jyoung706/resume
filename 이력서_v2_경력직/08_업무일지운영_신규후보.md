### A. 코어 프레임워크 (@orkis/core)

**N1. ComponentScan 정합성 — tsconfig exclude 미존중 + 설치본 스캔 차단**
- 출처: 연구노트 orkis/2026-06-11~12, 커밋 `92f705e`·`c232003`
- 상황: ComponentScan이 Backend tsconfig에서 제외한 파일까지 스캔, node_modules 전역 ignore가 설치본 `@orkis` extensions 스캔을 차단해 DI 빈 미등록.
- 원인: ① exclude 미존중 ② NODE_ENV 문자열 의존으로 dev/prod 판정 부정확.
- 해결: 런타임 ts-node가 로드한 tsconfig exclude를 스캔 ignore로 병합, dev판정을 TS 런타임 실측으로 교체, source/package 스캔 모드 분리.
- 결과: 데스크탑 빌드 제외 파일이 스캔에서 빠지고, 설치본 extensions 정상 스캔→DI 정상.
- 가치: 자체 프레임워크 깊은 이해 입증. **추천.**

### B. 통신 인프라

**N2. Socket reader 2단 백프레셔 — 무한 버퍼 누적 차단**
- 출처: 업무일지 5월1주차(2026-05-06)·연구노트 orkis/2026-05-06, 커밋 backend `b06f7ad`·desktop `58d899a`
- 상황: backend↔desktop 양방향 소켓에서 수신 처리 속도 불균형 시 reader buffer 무한 누적(메모리 누수 위험).
- 원인: backpressure 부재, 통신 상수가 양쪽에 분산.
- 해결: 공통 `socket/constants.ts`로 상수 단일화 + 2단 백프레셔(PAUSE 8MB → KILL 32MB), inflight/burst 카운터로 backlog 가시화.
- 결과: 정상 트래픽 영향 0, 폭주 시 데이터 무손실 차단.
- 가치: 상세본 L195의 "backpressure dispatch loop"보다 구체적(임계값 2단). **추천(상세본 통신 섹션 보강).**

### C. 데스크탑/백엔드 chat (어댑테이션 범위)

**N5. Chat 에러 종료 UI 복구 — step error backfill + SSE 응답 누수**
- 출처: 업무일지 6월2주차(2026-06-10~11), 커밋 backend `1a75b54`·`5e16c16`
- 상황: 에러로 멈춘 채팅이 재조회 시 "75% 생성중"으로 고착, 중간 토큰이 답변처럼 저장, SSE가 안 닫혀 origin당 6연결 한도 누적(7번째 채팅부터 pending).
- 원인: ① 비정상 종료 시 진행 step 미영속화 ② 에러 문구 대신 중간 토큰 저장 ③ 모든 에러 경로에서 SSE 응답 미종료.
- 해결: 종료 시 진행 step을 error로 backfill+실시간과 동일 에러 문구 영속화, 전 에러 경로에 SSE 종료 보장.
- 결과: 재조회가 실시간과 동일 렌더, SSE 정상 해제.
- 가치: 스트리밍 안정성·리소스 누수. 단 chat 도메인이라 "어댑테이션" 톤 유지 필요. **조건부 추천.**

**N6. RAG 전처리 상태 SSOT — AI가 직접 SQLite status UPDATE**
- 출처: 업무일지 5월3주차(2026-05-20), 커밋 backend `0affe69`·desktop-ai `64a2080`(상세본 L240에 커밋만 언급)
- 상황: Backend가 status를 먼저 `processing`으로 단정 UPDATE→AI가 중간에 죽으면 DB가 `processing`에 고착(실상태 불일치).
- 원인: 작업주체(AI)와 상태주체(Backend) 분리로 SSOT 부재.
- 해결: AI가 진행에 따라 `pending→processing→success/failed` 직접 UPDATE, Backend 사전단정·폴링 제거.
- 결과: 상태-실제 일치, 헤더 DB 아이콘 폴링 정상.
- 가치: 상세본에 커밋만 있고 케이스 설명 없음→짧게 보강 가능. **조건부 추천.**

**N7. SQLite DB 삭제 시 WAL + 전처리 산출물 정리**
