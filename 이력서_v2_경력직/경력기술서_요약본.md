## (주)범익 — Orkis (Text-to-SQL 연구 플랫폼)

자연어 질문을 LLM/RAG가 SQL로 만들어 **사용자가 등록한 DB에 실제 실행**해 결과를 돌려주는 연구 플랫폼. Cloud(웹)·Desktop(단일 머신)을 **하나의 코드베이스**로 지원. 본인은 백엔드·인프라를 본업으로 프레임워크 설계부터 데스크탑·k8s 배포까지 담당했다.

> **역할 경계** — SQL 생성·RAG 모델링과 **클라우드 채팅 SSE 파이프라인은 AI 연구원/타 개발자 담당**. 본인은 그 위의 백엔드 인프라·시스템 연결·배포를 맡고, 데스크탑은 Redis 없이 돌도록 어댑테이션했다.

### 1. `@orkis/core` — 자체 백엔드 프레임워크

**역할** 설계·구현 주도 · 기여도 약 90% · 115파일 · TS·Express

**왜 만들었나.** Nest·TypeORM은 '요청별 동적 커넥션(멀티테넌시)·멀티 DB 추상화'라는 Text-to-SQL 핵심 요구를 충족하지 못하는 데다, 불필요한 기능까지 포함해 무겁다. 리소스를 많이 쓰는 LLM 서버와 달리 **백엔드는 필요한 기능만 담아 작은 사양에서 가볍게 동작·수평 확장**해야 했기에, 핵심만 담은 **경량 프레임워크**를 자체 구현하기로 결정. 설계 전 **Spring Boot의 DI·트랜잭션 전파를 레퍼런스로 학습**(주력 스택이 아닌 Java/Spring 약 2개월 분석)해 핵심만 TS로 재해석.

**한 일과 결과.**
- JS에 없는 **트랜잭션 컨텍스트 전파**를 Proxy/Reflect로 자체 구현 → `@Transactional` 전파 정책(REQUIRED·REQUIRES_NEW·SUPPORTS) 확보, 접근 시점에만 clone하는 Lazy 전략으로 오버헤드 최소화
- **DB 5종**(PG·SQLite·MariaDB·Redis·File)을 단일 어댑터로 추상화 → Cloud/Desktop을 **같은 코드로** 지원
- 요청 userId로 런타임에 **사용자별 DB 커넥션 생성(멀티테넌시)** → "임의 DB에 SQL 실행"을 떠받치는 핵심 인프라
- 통합 테스트 30종 + 트랜잭션 전파 5시나리오를 PostgreSQL `xmin`으로 격리 검증

**대표 트러블슈팅.**
- **SQLite 동시 트랜잭션 충돌** — 단일 커넥션 공유 구조라 동시 `BEGIN`이 "transaction within a transaction"으로 깨짐. 비동기 mutex로 직렬화하고 rollback 실패 시 deadlock까지 방어 → 3병렬 호출 동시 200 확인.
- **성능 리팩토링이 부른 격리 회귀** — Bean clone 오버헤드를 줄이는 **성능 최적화(Lazy clone)** 도중, clone이 트랜잭션 래핑 전 원본을 snapshot해 `REQUIRES_NEW`가 우회됨. 에러 없는 격리 회귀를 주입 순서를 바로잡아 복원, `xmin`으로 두 서비스가 독립 트랜잭션(1299/1300)임을 입증.

**한 일과 결과.**
- 네이버·카카오·구글 **OAuth**를 단일 오케스트레이터로 통합(bcrypt·Bearer 병행)
- 사용자별 DB 런타임 등록 + 접속정보 **AES-256-GCM 암호화**, PG·MySQL/MariaDB·SQLite 타입별 분기
- **단일 코드베이스로 Cloud·Desktop 동시 지원** — TS `paths` fallback + `tsc-alias`로 같은 import를 빌드 타깃별로 자동 해석
- 멀티스테이지 Docker(Cloud/Desktop 분리)·k8s 무중단 롤링 배포

**대표 트러블슈팅.**
- **exclude가 transitive import를 못 막는다** — 클라우드 빌드에 desktop 코드가 간접 의존으로 섞임. tsconfig 4단 분리 + 폴더 단위 cascade exclude로 차단, prod 산출물 경량화.
- **데스크탑 채팅 어댑테이션** — Redis 없는 데스크탑에서 AI SSE를 직접 구독해 Cloud와 동일 이벤트 계약으로 변환 → 프론트 코드 한 줄도 안 바꾸고 통합.

### 4. `orkis-desktop` — Electron 데스크탑 런처

**역할** 설계·구현 주도 · 기여도 약 95%

**한 일과 결과.**
- Electron 39 + **Podman 컨테이너 런처**, mac/Windows(NAT·mirrored) 크로스플랫폼 네트워킹을 단일 함수로 추상화
- `podman events` 스트림으로 서비스 헬스를 IPC push(이벤트 기반 ready 판정)
- **통신 인프라를 4단계로 진화**(UDS → Podman → TCP 자체 프로토콜 → 표준 HTTP/SSE), 자체 소켓 인프라 전면 제거

**통신 진화 의사결정.** 빠른 IPC(UDS) → 운영·보안(컨테이너) → 컨테이너 제약 회피(TCP) → 유지보수성·표준화(HTTP/SSE). 매 단계 "무엇을 얻고 무엇을 버리는가"를 명시적으로 판단.

**대표 트러블슈팅.**
- **소켓에서 깨지는 한글 + 무한 버퍼** — chunk 경계 UTF-8 손상·크기 가드 부재로 메모리 누수. length-prefix wire format(4바이트 헤더) + 4MB 상한으로 한글 무손상·악성 프레임 차단.

### 5. `orkis-desktop-ai` — 클라우드 AI 디커플링·이식

**역할** 기여도 약 90% · *비즈니스 로직은 재사용, 새 언어(Python)를 학습해 적용*

**무엇을 했나.** 기존 클라우드 AI(`orkis-ai`)가 **Ray(분산 액터)·Redis에 강결합**돼 있던 것을, Ray/Redis 없는 **데스크탑 단일 머신에서 동작하도록 분리·이식**. LangGraph 비즈니스 로직은 원본을 그대로 재사용하고 통신·동시성·저장 계층만 교체.

| 클라우드 (강결합) | 데스크탑 (이식) |
|---|---|
| Ray Actor 6종 (분산 워커) | asyncio Task + TaskManager 추상화 |
| Redis XADD 이벤트 스트림 | asyncio.Queue + SSE 스트리밍 |
| PostgreSQL / Redis | SQLite / 파일 |

**결과.**
- **단일 uvicorn 프로세스로 통합** → 배포·포트·디버깅 단순화, 비즈니스 로직 100% 재사용
- **Ray Actor 예약 메모리(약 3.1GB)·Ray 런타임·Redis를 제거**해 상주 메모리 **약 60% 경량화(약 4.5GB → 2GB)** *(제거한 구성요소 기준)*

**대표 트러블슈팅.**
- **동기 I/O가 이벤트 루프를 막음** — read loop가 동기 파일·DB I/O를 직접 실행. `create_task` 백그라운드화 + `asyncio.to_thread` 오프로드 → 100 동시 채팅 3.2s→2.4s *(과거 업무일지 측정값)*.
- **전송 계층 추상화** — `ContextVar` transport가 task 경계에서 유실되던 문제를 `EventTransport` ABC로 추상화 → 이후 SSE 전환에 그대로 재사용.

### 1. 그룹웨어 채팅 — 서버·클라이언트 풀스택

**역할** 요구분석~설계~구현~배포 주도 · 기여도 서버 약 95% · 클라 약 65%

**왜 했나.** 회사(테넌트)별로 격리된 다중 조직이 한 클러스터 위에서 동작해야 하는 멀티테넌시 메신저가 필요했다.

**한 일과 결과.**
- **PM2 cluster + Redis 어댑터**로 다중 워커 간 메시지 브로드캐스트 → 수평 확장 가능한 소켓 서버
- 회사별 **namespace로 멀티테넌시 격리**(userId/companyId 이중 인증)
- MongoDB(메시지) + MySQL(사용자) 하이브리드, **JWT 만료 자동 갱신**으로 연결 끊김 없음
- Vue 3 + Pinia로 다중 namespace 동시 연결, Electron으로 **mac/Windows 크로스플랫폼 빌드**

**대표 트러블슈팅.**
- **PM2 cluster에서 cron 중복 실행** — 워커마다 예약작업이 각각 실행됨. Redis `SET NX PX` 분산 락으로 단일 인스턴스만 실행, TTL로 데드락 방지 → 워커 수와 무관하게 정확히 1회 실행.
- **namespace 중복 연결** — 서버 `Set` dedup + 클라이언트 `socketInit` 가드로 소켓을 1:1로 유지.

### 2. 병원 발렛파킹 주차관제 시스템

**역할** 백엔드 주도 · 기여도 약 80% · REST 119개

**한 일과 결과.**
- **Layered 3계층**(Controller / Service / DAO)으로 119개 API를 일관되게 구현
- **HTTP·WebSocket을 같은 서버·로직으로 통합** → 한 번의 입·출차 처리로 DB·실시간 화면·Web Push가 함께 동기화
- 소켓 JWT 인증 + 중복 로그인 강제 종료, TypeORM 다중 데이터소스(메인 + SMS)

**대표 트러블슈팅.**
- **중복 로그인 시 유령 세션** — `connectedUsers` 맵으로 세션을 추적, 신규 로그인 시 기존 소켓을 `disconnect(true)`로 강제 종료. HTTP 로그인 흐름이 WebSocket 세션을 직접 제어하도록 통합.
