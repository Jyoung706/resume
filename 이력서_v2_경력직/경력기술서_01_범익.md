## (주)범익 — Orkis (Text-to-SQL 연구 플랫폼)

> **기간** 2025.02.17 ~ 재직 중 (약 1년 4개월) · **소속** 기업부설 연구소 / 연구원
> **역할** Server Engineer · Infra Engineer · 자체 백엔드 프레임워크 설계자

### 프로젝트 개요

자연어 질문을 입력하면 **LLM/RAG 서버가 SQL을 생성하고, 사용자가 등록한 DB에 실제로 실행해 결과를 반환**하는 Text-to-SQL 연구 플랫폼 **"Orkis"**. 클라우드(웹) 버전과 데스크탑(단일 머신) 버전을 **하나의 코드베이스**로 동시 지원하며, 7개 저장소로 구성된다.

**본인의 역할 범위**는 백엔드/인프라를 본업으로, ① 자체 백엔드 프레임워크 `@orkis/core` 설계·구현(주도), ② 백엔드 앱의 인증·동적 DB·단일 코드베이스 빌드·배포(주도), ③ 백그라운드 작업 서비스 `orkis-jobs`(단독에 가까움), ④ Electron 데스크탑 런처 `orkis-desktop`(단독 소유), ⑤ 클라우드 AI 백엔드를 데스크탑용으로 디커플링·이식한 `orkis-desktop-ai`(단독에 가까움), ⑥ 연구소 쿠버네티스 플랫폼 `k8s-config` GitOps 구축(약 80% 주도)이다.

- **역할 경계 명시** — SQL 생성·RAG 모델링, 그리고 **클라우드 채팅/RAG SSE 파이프라인(Redis 스트림 폴링 기반 `FetchStreamService`)은 AI 연구원/타 개발자 담당**이다. 본인은 그 위에서 백엔드 인프라·시스템 연결·배포를 담당했고, 데스크탑에서는 Redis 없이 동작하도록 **어댑테이션**(프론트 무수정·동일 이벤트 계약)하는 수준으로 관여했다. AI 연구원·프론트엔드 개발자와는 **이벤트 계약(orkis-interface 타입 패키지) 기반으로 협업**하며, 서버를 양측의 접점으로 설계했다.

## 1. 자체 백엔드 프레임워크 `@orkis/core`

### 개요 · 역할

Node.js/TypeScript 환경에서 **Spring Boot 스타일의 데코레이터 기반 개발**을 가능하게 하는 프레임워크를 직접 설계·구현했다. 성숙한 ORM/프레임워크(Nest·TypeORM)가 **요청별 동적 커넥션(멀티테넌시)·멀티 DB 추상화**라는 Text-to-SQL 핵심 요구를 충족하지 못해, 필요한 핵심만 자체 구현하기로 결정한 것이 출발점이다.

- **규모** **305커밋 / 7,658 LOC / 115 TS파일 / 데코레이터 30종 / 5종 DB 어댑터**. v1.0.30+ GitLab Package Registry 배포(yalc 개발 워크플로).
- **역할** 단독 설계·구현(주도, 메인). 메타데이터 저장(데코레이터)과 인스턴스화·주입(BeanResolver)을 책임 분리하는 구조부터 직접 잡았다.

### 주요 구현

- **DI 컨테이너** — `@Controller`/`@Service`/`@Dao`/`@Component`/`@Autowired` + ComponentScan 자동 빈 등록. 순환참조 감지(BeanResolver), 싱글톤/프로토타입 스코프, `@InjectConnection(name, {dynamic})`, `@Value(env)`. *왜* — 의존성 배선을 선언적으로 만들어 멀티 DB·동적 커넥션을 일관된 방식으로 주입하기 위함.
- **`@Transactional` 전파** — REQUIRED / REQUIRES_NEW / SUPPORTS 전파 정책을 Symbol 기반 메타데이터로 구현(`transactionResolver.ts` 628 LOC). JavaScript에 없는 *트랜잭션 컨텍스트 전파*를 Proxy/Reflect 메타프로그래밍으로 풀고, **실제 접근 시점에만 Bean을 clone하는 Lazy Proxy 전략**(2+depth Bean만 clone)으로 오버헤드를 최소화했다.
- **동시성 제어** — 진행 중인 connection 획득 Promise를 Map으로 관리해 동시 획득을 직렬화(중복 생성 race 방지), 다중 connection 트랜잭션의 부분 실패를 추적(commit/rollback)한다.
- **멀티 DB 추상화** — `BaseAdapter<TConn, TConfig>` 하나로 PostgreSQL/SQLite/MariaDB/Redis/File을 통일. `?` placeholder를 DB별로 정규화하고, `supportsTransaction()`로 어댑터별 트랜잭션 지원을 분기한다.
- **동적 DB 연결(멀티테넌시)** — `DynamicConnectionSupport` / `prepareDynamicDBConnection()`로 **요청의 userId로 런타임에 사용자별 connection을 생성**한다. Text-to-SQL의 "사용자가 등록한 임의 DB에 SQL 실행" 요구를 떠받치는 핵심 인프라.
- **HTTP 클라이언트 / 스케줄러 / 인터셉터 / SSE** — undici 2계층(global Agent / Pool) keep-alive·인터셉터 체인·Unix socket HTTP 클라이언트; node-cron 4.x 기반 `@Job`(cron|interval discriminated union)·graceful shutdown(30s drain)·pause/resume/trigger 런타임 제어; 4 point-cut(FILTER/BEFORE/AFTER/EXCEPTION) 인터셉터; `@SSEConnection` + `SSEStream`(onClose) 유틸. 파일 업로드 path-traversal 방어, CORS·로깅·글로벌 에러 핸들러, k8s 환경 대응.
- **검증 인프라** — 멀티 DB·스케줄러·동적 커넥션 통합 테스트/예제 30종, 트랜잭션 전파 5개 시나리오를 PostgreSQL `xmin` 기반으로 격리 검증.

### 트러블슈팅 (STAR)

#### 1-1. SQLite 동시 트랜잭션 충돌 — 자체 비동기 mutex로 직렬화 ★

- **상황** 데스크탑 첫 화면 로드 시 codes/keywords/llm 등 여러 API가 병렬 호출됨.
- **원인** SQLite 어댑터의 `connect()`는 **단일 connection을 공유**(PostgreSQL은 Pool에서 매번 새 client 반환). 동시 요청이 같은 connection에 각각 `BEGIN`을 실행 → `cannot start a transaction within a transaction`, 연쇄적으로 `no transaction is active`까지 발생.
- **해결** 어댑터에 **비동기 mutex(`txMutexHeld` + waiter 큐)**를 직접 구현해 트랜잭션을 직렬화. BEGIN 실패 시 즉시 해제, rollback 실패에도 `finally`로 해제(누락 시 영구 deadlock)하는 엣지 케이스까지 방어. WAL 모드 적용.
- **결과** 동시 BEGIN 충돌 제거, 3개 병렬 호출 동시 200 검증. **"어댑터마다 connection 수명주기가 다르다"는 근본 원인**까지 파고들었고, 주석에 "병목 실측 시 PG식 pool 전환"이라는 향후 개선까지 명시.

#### 1-2. 리팩토링이 부른 트랜잭션 격리 회귀 — `REQUIRES_NEW`가 우회됨 ★

- **상황** Bean clone 성능 리팩토링(`CLONE_SPEC`) 직후, `REQUIRES_NEW`가 기존 트랜잭션을 새로 분리하지 않고 그대로 올라타는 회귀 발생(에러는 없음).
- **원인** `buildCloneSpec`이 `wrapTransactionalMethods`보다 **먼저** 실행되어, clone이 트랜잭션 래핑 전 원본 메서드를 snapshot → propagation 분기 로직을 통째로 우회.
- **해결** `injectDependencies` 순서를 `wrapTransactionalMethods → buildCloneSpec`으로 재배치하고, `wrapTransactionalMethods(hasConnections)` 파라미터와 `beanClone[CLONE_SPEC]` 주입으로 재진입 안정성 확보.
- **결과** PostgreSQL **`xmin` 기반 검증**으로 두 서비스가 각각 독립 트랜잭션(xmin 1299/1300)임을 확인 — 회귀 당시 단일 공유(1298)와 대비해 격리 복원을 입증.

#### 1-3. 요청 스코프 트랜잭션 — 전역 Map이 만든 충돌·누수

- **원인** connection 획득 Promise를 **모듈 전역** `Map<requestUUID:datasource, Promise>`로 관리 → ① 8자 UUID Birthday collision로 다른 요청이 같은 Promise를 공유할 위험, ② `connect()` 실패 시 rejected Promise가 Map에 영구 잔존해 재시도 불가 + 메모리 누수.
- **해결** `pendingConnections`를 전역에서 **요청별 `createTransactionContext` 내부 로컬 변수**로 이동하고, clone/proxy/adapter 시그니처에 인자로 전파. 각 요청이 독립 맵을 소유하고 스코프 종료 시 자동 정리.
- **결과** 트랜잭션 격리를 구조적으로 보장. REQUIRED/REQUIRES_NEW/SUPPORTS 5개 시나리오를 SQLite placeholder 계약으로 검증하는 테스트 인프라까지 구축.

#### 1-4. 의존 라이브러리 버그·dialect 차이 우회 (묶음)

- **SQLite `RETURNING` 더블 실행** — `sqlite3` npm v5의 `db.all()`이 `INSERT/WITH ... RETURNING`에서 statement를 두 번 실행하는 알려진 버그. RETURNING 포함 시 `db.each()`(streaming step)로 우회하는 `executeReturning` 분기 추가, `\bRETURNING\b` word-boundary 정규식으로 컬럼명·주석 오탐 방지.
- **MariaDB `BigInt` → `JSON.stringify` throw** — 어댑터 추가 시 드라이버가 `BIGINT`/`COUNT(*)`를 JS `BigInt`로 반환해 직렬화가 throw. 어댑터 기본값 `bigIntAsNumber: true`로 차단(단 `decimal`은 정밀도 보존 위해 제외), pool/단일 구성을 discriminated union으로 타입화.
- **DB 연결 실패는 "크래시"로 취급** — Pool 모드에서 연결 가능 여부가 첫 쿼리까지 지연되는 문제를, `PostgreSQLAdapter.create()`에서 `connect()+release()`로 즉시 검증하고 실패 시 `writeCrashLog()` → `process.exit(1)`로 처리. "절반만 살아있는" 프로세스 제거.
- **부트스트랩 로그 유실 / 라우터 슬래시** — env 로드 전 `LOG_LEVEL` 공백 문제를 System/Application 로거 분리로(코어 로그는 레벨 무관 출력) 해소. Controller path 앞뒤·연속 슬래시 조합 실패를 `normalizePath()`로 정규화.

### 성과

JS 생태계에 없던 트랜잭션 컨텍스트 전파를 프레임워크 수준에서 구현하고, 동시성·dialect 차이가 만드는 "조용한 버그"(에러 없는 빈 화면·메모리 누수)를 라이브러리 소스·DB 드라이버 동작까지 추적해 근본 원인으로 해결했다. 이 프레임워크가 백엔드 앱·jobs·데스크탑을 떠받치는 공통 기반이 되었다.

## 2. 백엔드 앱 `orkis-backend` — OAuth · 동적DB · 단일 코드베이스 빌드 · 배포

### 개요 · 역할

`@orkis/core` 위에서 동작하는 백엔드 애플리케이션(32,906 LOC / 177파일). GitHub는 스냅샷 1커밋이라 커밋 수가 무의미해 **LOC·아키텍처·코드 구조 교차검증으로 주도(메인)를 입증**한다. 단, **클라우드 채팅/RAG SSE 파이프라인(`FetchStreamService`, Redis XREAD 폴링)은 타 개발자 담당**이므로 본인 성과에서 제외하고, 아래 영역만 본인 주도로 기술한다.

### 핵심 구현 (본인 주도 — 코드 구조 근거)

- **OAuth 3사 통합** — 네이버·카카오·구글을 `AuthService`(약 530 LOC) 단일 오케스트레이터로 통합. `parseOAuthUserInfo()`로 제공사별 응답을 정규화하고, authorization code → token → userInfo → 신규/기존 분기. bcrypt 패스워드 로그인 병행, Bearer 토큰 인증.
- **동적 DB 연결 적용** — 코어의 `DynamicConnectionSupport`를 `DbConnectionService`에 적용. 사용자별 DB를 런타임 등록하고, 접속정보를 **AES-256-GCM으로 암호화**(`encryptPassword`/`decryptPassword`), PostgreSQL·MySQL/MariaDB·SQLite 타입별 어댑터로 분기, `DbConnectionChecker`로 사용자 DB liveness 측정.
- **단일 코드베이스 cloud/desktop 빌드** — `tsconfig.prod`(`@/* → src/main`) vs `tsconfig.desktop`(`@/* → [src/desktop, src/main]` fallback) + `tsc-alias` 후처리로, **같은 import가 빌드 타깃에 따라 desktop override 우선 / 없으면 main fallback**으로 자동 해석되게 설계. desktop은 DAO만 override하고 Controller/Service는 재사용. 클라우드 개발자의 IDE에는 desktop 코드가 보이지 않도록 격리.
- **데스크탑 채팅 어댑테이션** — (파이프라인 설계가 아니라) 단일 코드베이스 맥락에서 **데스크탑이 Redis 없이 AI SSE를 직접 구독**하도록 어댑테이션. 자세한 내용은 1-5 트러블슈팅 참조.
- **배포** — 멀티스테이지 Dockerfile(Cloud/Desktop 분리 빌드), k8s 무중단 롤링(`maxSurge:1 / maxUnavailable:0`), PM2 클러스터 + PG Pool 검증.

### 트러블슈팅 (STAR)

#### 2-1. "exclude가 transitive import를 막지 못한다" — 폴더 단위 cascade exclude ★

- **상황** Jenkins 빌드가 `paths를 못 찾음` / 단일 tsconfig으로 dev·prod를 모두 빌드해 prod 산출물에 sourceMap까지 포함되는 문제. 클라우드 빌드에 desktop 코드가 섞여 들어옴.
- **원인** tsconfig `exclude`가 명시 파일은 빼더라도 **transitive import(간접 의존)는 막지 못하는** 비자명한 컴파일러 동작. 추가로 `.gitignore`의 단어형 패턴이 도메인 폴더까지 매칭해 소스가 누락됨.
- **해결** tsconfig를 base/dev/prod/fallback **4단 분리**하고 **폴더 단위 cascade exclude**로 desktop 코드 유입을 차단, `build:dev`/`build:prod` 스크립트와 `rootDir`를 명시(ts5011 해소). `.gitignore` 패턴을 슬래시 prefix로 한정하고 폴더명을 단순 명사로 정리.
- **결과** Cloud/Desktop 빌드 산출물·ComponentScan·런타임 resolution을 모두 검증, prod 산출물 경량화. 컴파일러 내부 동작 이해가 필요한 아키텍처 작업.

#### 2-2. 데스크탑 채팅 어댑테이션 — Redis 없이 AI SSE 직접 구독 (프론트 무수정)

- **상황** Cloud 채팅은 `AI → Redis stream → 폴링 → SSE`의 3-hop 구조(이 파이프라인 자체는 타 개발자 담당). 데스크탑에는 Redis가 없어 그대로 쓸 수 없음.
- **해결** 데스크탑 백엔드가 AI `/chat/start` SSE를 **직접 구독**(fetch ReadableStream + TextDecoder로 `\n\n` 단위 파싱)한 뒤, Cloud와 **동일한 `message_stream {chatId, type, payload}` 계약**으로 변환해 프론트에 forward. 취소 채팅의 완료 코드 충돌(사용자 취소 9001을 SQL 실패 9003이 덮어쓰던 문제)은 `if (finalCode !== 9001)` 가드로 코드를 보존하고 9005/9006을 9003으로 롤업해 Cloud와 범위 통일(9001~9004).
- **결과** 완전히 다른 두 전송 구조를 동일 이벤트 계약으로 추상화해 **프론트 코드를 한 줄도 바꾸지 않고** 통합. 취소된 대화도 "중지됨" 상태로 이력 복원.

#### 2-3. dialect별 반환 타입 차이 / placeholder 순서 (묶음)

- **SQLite `EXISTS`가 boolean이 아님** — `SELECT EXISTS(...)`가 SQLite에선 정수를 반환해 `rows[0]?.exists` 비교가 항상 false(LLM 모델 존재 조회 깨짐). `SELECT COUNT(1) as cnt` + `(cnt ?? 0) > 0`으로 교체.
- **PG↔SQLite placeholder 순서** — SQLite `?`는 SQL 등장 위치 순서에 의존하는데 인자를 PG식(`$N` 자유 순서)으로 넘겨 매칭 0건(공지 화면이 항상 "준비 중"). `?` 위치에 맞춰 인자 재정렬 후 전 DAO 회귀 검증, "동일 `$N` 2회 참조 시 인자 2번 push" 규칙으로 교훈 체계화.

### 성과

인증·멀티테넌시·단일 코드베이스 배포라는 백엔드 핵심을 주도하고, dialect 차이가 만드는 silent failure를 근본 원인까지 추적해 Cloud/Desktop을 한 코드베이스로 안정 운영했다.

## 3. 백그라운드 작업 서비스 `orkis-jobs`

### 개요 · 역할

세션 Redis 라이프사이클·아카이빙·헬스체크를 담당하는 스케줄링 서비스. **68커밋 / 3,342 LOC / 22파일, 단독에 가까움.**

### 핵심 구현

- **5개 `@Job` 클래스** — `AiServerHealthJob`(30s, 4xx 즉시실패 / 5xx·timeout 1회 재시도), `ChatSessionCleanupJob`(10s, 60분 타임아웃 강제 완료), `RedisCleanupJob`(5s, 완료 세션 삭제), `QuestionCountJob`(일 자정 리셋), `PreprocessProgressJob`.
- **회복탄력 Redis 인프라** — ioredis exponential backoff + jitter 재시도(최대 10회), 6종 이벤트 리스너, ping 헬스체크, 다중 커넥션 동시 초기화/graceful disconnect.
- **Redis 정리 최적화** — `SCAN`(논블로킹, batch 1000, `setImmediate`로 5000키마다 yield) + Pipeline(batch 50) + 실패 시 순차 폴백. 세션당 에러 격리로 이벤트 루프 블로킹 없이 처리.
- **세션 NAS 아카이빙** — `{chatroom_id}/YYYY-MM-DD.json` 형식. 트리거를 cron pull → **HTTP push**(`POST /archive/internal` + 큐잉)로 전환.
- **DI 마이그레이션** — 수동 `JobScheduler` / 정적 `DatabaseManager` **1,000줄+ 제거** → `@Job`/`@Autowired`/`@InjectConnection` 선언적 전환, 도메인 폴더 재편.

### 트러블슈팅 (STAR)

#### 3-1. Redis 세션 완료 판정 결함 — 메모리 누수

- **원인** 완료 세션 삭제 시 9개 필수 단계(`0_0`~`2_2`) 전체 존재 여부로 판정 → 단계 하나라도 누락되면 영구 잔류.
- **해결** 종료 단계 `2_2`(job_end) 존재만으로 판정하도록 단순화 + 빈 데이터 가드.
- **결과** 실제 종료 신호 기준으로 정확히 정리, 잔류 세션 제거.

#### 3-2. 배포되자마자 CrashLoopBackOff — `.gitignore`가 `resources/`를 통째로 무시

- **상황** k8s 배포 직후 jobs pod가 `[FATAL] Production mode requires environment file: /app/resources/prod.env`로 크래시.
- **원인** `.gitignore`의 `resources/` 규칙이 폴더를 통째 제외 → 이미지에 prod.env placeholder조차 없음. ConfigMap으로 env가 주입되는 k8s 환경인데 코어가 **파일 존재를 강제**.
- **해결** 이중 보강 — ① `.gitignore`를 `resources/dev.env`로 한정(폴더는 추적), ② **코어(1.0.22)에서 prod 강제 throw를 제거**해 파일 있으면 dotenv 로드 / 없으면 warn 후 진행, ConfigMap env는 `process.env` 우선성으로 사용. "신규 서비스마다 같은 회피책 반복"을 피하려 코어 정책을 완화하는 쪽을 선택.
- **결과** ConfigMap-only 운영 패턴 지원(`k8s-config`와 연계).

### 성과

레거시 1,000줄+를 선언적 DI로 걷어내며 운영 안정성과 가독성을 확보하고, jobs의 배포 실패를 코어 정책 차원에서 근본 해결했다.

## 4. Electron 데스크탑 런처 `orkis-desktop` (통신 마이그레이션 포함)

### 개요 · 역할

웹 Orkis를 Electron 39 + electron-vite 데스크탑 앱으로 패키징하고, backend/AI를 **로컬 Podman 컨테이너로 구동하는 런처**. **91커밋 / 순 +8,291 LOC, 단독 소유.**

### 핵심 구현

- **부트스트랩 프레임워크 `OrkisApplication`** — 라이프사이클 일원화, `ipcHandle<A,R>()` 제네릭, `safeOn()` 에러 격리, 통합 shutdown. config/container/ipc/health/services/errors 폴더 응집.
- **`PodmanEventsMonitor`**(109줄) — `podman events --format json` 스트림을 파싱해 `service:ready/unhealthy/down`을 IPC로 push, 2s backoff 재연결, 상태 dedup. 소켓 헬스 폴링을 대체.
- **크로스플랫폼 네트워킹** — mac bridge / Windows-NAT(vEthernet WSL IP) / Windows-mirrored(`.wslconfig` 정규식 감지 → `--network host`)를 `ContainerNetConfig` 단일 함수로 집중.
- **통신 인프라 마이그레이션** — 자체 TCP 소켓 인프라(8파일·wire format·shared secret)를 제거 → Podman HEALTHCHECK 이벤트 기반으로 전환. `podman build --format docker`로 OCI HEALTHCHECK 메타데이터를 보존.
- **세션 쿠키** — 수동 `http.request` 파싱(196줄) → Chromium `net.fetch` 자동 처리. Finder ENOENT(PATH miss) 절대경로 대응. dmg/pkg·nsis·AppImage 패키징, system tray, 플랫폼별 타이틀바(mac hiddenInset).

### 통신 방식의 진화 (의사결정 서술)

데스크탑은 **Electron 런처 · Node 백엔드 · Python AI** 세 프로세스가 한 머신 안에서 통신해야 한다. 이 통신 계층을 네 단계에 걸쳐 진화시키며 매 단계 트레이드오프를 명시적으로 판단했다.

1. **바이너리 + UDS** — Python AI를 PyInstaller 단일 바이너리로, 백엔드를 자식 프로세스로 spawn, 셋을 Unix Domain Socket으로 연결. *장점* 포트 점유 없는 빠른 IPC. *한계* OS·아키텍처별 빌드/서명/의존성 부담, 약한 런타임 격리.
2. **Podman 컨테이너** — 의존성·런타임을 이미지로 봉인해 운영 편의·보안·재현성 확보. *부작용* 컨테이너가 Podman machine(Linux VM) 안에서 돌아 **host의 UDS `.sock` 경로 공유 불가** → UDS 폐기.
3. **TCP loopback + 자체 wire format** — length-prefix(4바이트 헤더)·shared secret·backpressure dispatch loop를 직접 구현. *한계* 단방향 스트리밍(LLM 토큰)에 양방향 소켓을 쓰는 비대칭, 자체 프로토콜이라 디버깅에 tcpdump 필요, 양 끝단 스택 어긋남.
4. **표준 HTTP REST + SSE** — 결정타는 **Python AI의 소켓 read loop 블로킹**(→ 5-1). 자체 통신 인프라를 통째로 걷어내고 표준화. `socketClient.ts`/`socketSender.ts`/`EVENT_SOCKET_SECRET`을 완전 제거하고 Cloud 버전과 컨벤션 통일.

### 트러블슈팅 (STAR)

#### 4-1. 소켓 통신에서 깨지는 한글 + 무한 누적되는 버퍼

- **상황** 런처 ↔ 백엔드/AI 소켓 통신에서 한국어 등 UTF-8 멀티바이트가 chunk 경계에서 손상되고, 거대 메시지 수신 시 버퍼가 무한 누적(메모리 누수).
- **원인** LDJSON(`\n` split) 포맷은 멀티바이트가 chunk 경계에 걸리면 잘리고, 크기 가드가 없어 오염 메시지가 버퍼를 무한 점유.
- **해결** **length-prefix wire format** 도입 — `struct.pack(">I", len(body))` Big-Endian 4바이트 헤더 + UTF-8 body. 헤더 선읽기로 본문 길이만큼 `readexactly`, `MAX_MSG_BYTES`(4MB) 초과 시 `socket.destroy()`. `encodeFrame` 헬퍼로 server/backend/ai를 일원화.
- **결과** 한글 무손상 도달 + 50MB declared frame 즉시 차단 검증. (이 자체 포맷은 이후 HTTP/SSE 전환으로 최종 제거 — 4단계 진화의 일부)

#### 4-2. Windows 크로스플랫폼 컨테이너 런처

- **과제** Podman 컨테이너를 mac과 Windows(WSL2) 모두에서 안정 구동.
- **해결** mac / Windows-NAT / Windows-mirrored 3개 모드를 `ContainerNetConfig` 단일 함수로 집중(.wslconfig 정규식 감지, WSL 어댑터 IP 추출). `net.createServer().listen()`으로 포트 사전 확보, NSIS 설치 스크립트의 프로세스 종료 대기·APPDATA 경로 해결.
- **결과** `machine.manager`를 191→120줄로 축소하며 Windows 전용 workaround 제거.

#### 4-3. edge-triggered 이벤트를 level처럼 다루기 (묶음)

- **`app:ready` 유실** — 메인이 `client:connected` 직후 `app:ready`를 보내는데 F5 리로드 중인 Renderer가 놓침. `servicesReady` 상태를 두고 `did-finish-load`에서 재전송 + preload가 캐싱해 `onAppReady()` 호출 시 즉시 콜백.
- **상태 아이콘 회색**(프론트 연계) — `podman events`는 edge-triggered라 healthy 후 새로고침하면 store가 비어 RAG 아이콘이 회색. zustand `persist`로 마지막 상태 복원. → edge vs level 이벤트 모델의 근본 차이를 진단한 동일 계열.

### 성과

유지비 높은 자체 통신 인프라(UDS→TCP)를 표준 HTTP/SSE로 대체하고, 컨테이너 런처를 크로스플랫폼으로 안정화해 데스크탑 배포를 단독으로 책임졌다.

## 5. 클라우드 AI 백엔드 디커플링·이식 `orkis-desktop-ai`

### 개요 · 역할

기존 클라우드 AI(`orkis-ai`)가 **Ray(분산 액터)·Redis에 강결합**돼 있던 것을, 데스크탑(단일 머신, Ray/Redis 없음)에서 동작하도록 **디커플링·이식**한 작업. **54커밋 / 순 +14,131 LOC, 단독에 가까움.** 비즈니스 로직(LangGraph 워크플로)은 **원본을 재사용**하고, 통신·동시성·저장 계층만 단일 머신 환경에 맞게 교체했다. **새 언어(Python·asyncio)를 학습해 이식·통합한 작업이며, Python 숙련을 주장하지는 않는다.**

### 디커플링 대조 (파일 근거)

| 클라우드(orkis-ai, 강결합) | 데스크탑(orkis-desktop-ai, 이식) |
|---|---|
| Ray Actor 6종(Conversation/Preprocess/LLM/Embedding/Faiss 등, `ray_config.py` 예약 메모리 합 3,122MB) | `core/task_manager/chat.py` asyncio Task + `BaseTaskManager` ABC, `LocalLLMActor._RemoteCallable`로 `.remote()`를 로컬 async로 변환 |
| Redis XADD 이벤트 스트림 | `app/transport/sse_transport.py` asyncio.Queue + `EventTransport` ABC(`core/transport/base.py`) + `AsyncTransportStreamHandler` |
| PostgreSQL / Redis | SQLite / 파일(`app/preprocess/history_dao.py`, vector_search sqlite repository) |

- 3계층 — controller(FastAPI+Pydantic DTO, request-scoped `SSEStreamTransport`+ContextVar) / service(프로토콜 독립, LangGraph 실행) / repository.
- asyncio 최적화 — FAISS 검색 `asyncio.to_thread()` 오프로드, schema/data 전처리 `asyncio.gather` 병렬, `ChatTaskManager`(취소 가능 태스크). FastAPI lifespan warmup, Dockerfile HEALTHCHECK, uvicorn CLI를 PID1로(시그널 처리).
- 주요 커밋 — `6f50b54`(태스크매니저), `d2f3bcd`(TCP socket 제거→SSE), `64a2080`(SQLite 직접 UPDATE), `482c21d`(cloud FaissPreprocessService 재사용), `999ce64`(Transport ABC), `6b52d98`(gpt-5 캐시키 오염 버그).

### 트러블슈팅 (STAR)

#### 5-1. 동기 I/O가 이벤트 루프를 막는다 — read_loop 블로킹 ★

- **원인** Python AI의 read_loop에서 `await handler()`가 동기 I/O(파일 저장·DB 조회)를 그대로 실행 → 이벤트 루프를 점유해 동시 채팅 처리 성능 저하. (4단계 HTTP/SSE 전환의 결정적 계기)
- **해결** dispatch를 `create_task(_safe_dispatch())`로 백그라운드화해 read_loop 논블로킹 보장. 동기 I/O는 `asyncio.to_thread()`로 오프로드(`_load_context`/`build_worker`/`_save_context`). `readline()`으로 UTF-8 경계 잘림도 해소.
- **결과** 100개 동시 채팅 테스트에서 평균 **3.2s → 2.4s(약 25% 개선)**.

#### 5-2. asyncio 태스크에서 사라지는 transport (None) — 전송 계층 추상화

- **원인** `ContextVar`로 관리하던 transport가 `asyncio.create_task()` 시 자동 복사되지 않아 송신 시점에 None.
- **해결** `ContextVar` → 모듈 변수 기반 transport 컨텍스트(`init/get/set_transport`)로 전환하고, `client.send()` 직접 호출을 `transport.emit(...)`으로 추상화. `EventTransport` ABC로 소켓 외 전송 계층 교체를 가능하게 설계.
- **결과** 비동기 태스크 경계의 transport 유실 제거 + **이후 SSE transport 교체 시 이 추상화가 그대로 재사용됨.** (디커플링 설계의 핵심 — UDS/Named Pipe/TCP fallback chain을 통합 HTTP/SSE로, asyncio.Queue 기반 `SSEStreamTransport`로 수렴)

#### 5-3. 전처리 병렬화 / gpt-5 캐시키 오염 (묶음)

- **전처리 병렬화** — RAG 전처리(rag_type=ALL)가 schema→data를 순차 실행하던 것을 `asyncio.gather(..., return_exceptions=True)`로 병렬화(단위 작업 `_run_single_preprocess` 추출). 샘플 DB(290+10,609 docs) 6회 측정 **7.86s → 6.47s, 1.21x**(이론 최대 1.23x의 98%).
- **케이스 컨벤션 통일** — 백엔드(snake)↔AI(camel) 불일치로 매 호출 toCamel/toSnake 변환 → 매핑 누락 시 undefined 회귀 위험. AI Pydantic DTO를 camelCase로 통일하고 백엔드 `caseUtil.ts`(33줄)·변환 9곳을 전수조사 후 삭제("외부 컨트랙트=camel, 내부=snake" 경계 정립).

### 성과 (메모리 — 소스 기반 추정, 실측 아님)

Ray/Redis 의존을 제거하고 다중 프로세스(Ray head + N워커 + Redis)를 **단일 uvicorn 프로세스로 통합**해 배포·포트·디버깅을 단순화했다. 비즈니스 로직은 100% 재사용했다.

- **상주 메모리 약 4.5GB → 2GB(약 60%) 절감 추정** — *근거*: orkis-ai 추정 상주 = Ray Actor 예약 3,122MB(`ray_config.py` MEMORY_MB) + Ray 런타임(GCS/object store) 500~800MB + Redis 100~300MB ≈ 4.0~5.5GB; orkis-desktop-ai 추정 = uvicorn ~150MB + TaskManager/Queue/httpx ≈ 1.5~2.5GB. **실측이 아닌 설정값·일반 메모리 특성 기반 추정이며 런타임 패턴에 따라 변동한다.**

## 6. 쿠버네티스 플랫폼 GitOps 구축 `k8s-config`

### 개요 · 역할

연구소 인프라 전체를 **매니페스트로 코드화(IaC)**한 저장소. KT Cloud ManagedKS 위에 클러스터·네트워킹·스토리지·CI/CD·데이터 서비스를 직접 설계·구축. **98/123커밋(약 80% 주도)** — AI/전처리(ai-service·preprocess) 매니페스트 일부는 동료와 분담. `base/` `infrastructure/` `cicd/` `applications/` 구조로 도메인 분리. 온프레미스(E3TS)에서 GitLab·Jenkins를 직접 구축한 경험을 클라우드 네이티브로 확장했다.

### 핵심 구현

- **노드 역할 분리** — `nodeSelector` 라벨로 **노드풀 5종**(ingress/jenkins/gitlab/orkis-database/orkis-ai) 분리해 진입점·CI·DB·앱 자원 간섭 차단. metrics-server, `system-cluster-critical` 우선순위, 노드 장애 시 30초 빠른 복구 toleration.
- **무중단 진입 계층** — ingress-nginx v4.15.1 **DaemonSet + hostNetwork**로 전용 노드에서만 80/443 listen, KT L4 LB 패스스루, proxy-body-size 2g. cert-manager v1.20.2 + Let's Encrypt(prod/staging ClusterIssuer, HTTP-01)로 orkis.kr·git.orkis.kr·registry·jenkins TLS 자동 발급·갱신(staging 선검증 → prod 전환으로 rate limit 회피).
- **SSE 전용 인그레스** — `/api/sse → /sse` rewrite, 쿠키 Sticky Session(`ORKIS_SSE_AFFINITY` 86400s), `proxy-buffering off`, read/send timeout 1800s로 장시간 LLM 스트리밍 유지.
- **스토리지** — NFS subdir provisioner(NAS 172.25.1.129)로 동적 PVC화, **StorageClass 4종**(nfs-retain-prod / nfs-delete-dev / nfs-static / local-postgres), 13 PV·15 PVC(~1,184Gi).
- **CI/CD 자체 호스팅** — GitLab EE 18.0.6 on k8s(git.orkis.kr), Jenkins 2.541 LTS+JDK21(Kubernetes Plugin으로 **빌드마다 Agent Pod 동적 생성→소멸**). **크로스 네임스페이스 최소 권한 RBAC**(jenkins-agent@cicd → jenkins-deployer@orkis)로 배포 권한 격리.
- **데이터 서비스** — PostgreSQL 17-alpine(fsGroup 999, 로컬 PV), Redis 7-alpine **3종**(chat/stage/session)을 DB 전용 노드 배치. CVE 대응 업그레이드 이력(GitLab 2025-09, Jenkins 2026-04).
- **백업/DR 3종** — ① PG `pg_dumpall` 주간(토 18:00UTC, 14일 보존) ② GitLab 전체 분기 1회(STRATEGY=copy, 90일 압축/365일 삭제) ③ **GitLab 미러 일간**.

### 트러블슈팅 (STAR)

#### 6-1. 토큰을 디스크에 남기지 않는 GitLab 미러 백업 (보안 + DR) ★

- **과제** 전 사내 소스를 매일 자동 백업하되, NAS에 저장되는 백업물에 인증 토큰이 노출되지 않아야 함.
- **해결** CronJob에서 GitLab API를 페이지네이션으로 순회해 전 프로젝트를 증분 `git fetch`. 인증은 `oauth2:TOKEN`을 **fetch 호출 URL에만** 실어 보내고, **디스크의 origin URL에는 토큰 없는 clean URL**을 기록. 일요일엔 tar.gz 스냅샷 후 30일 초과분 자동 삭제.
- **결과** 소스 유실 시 임의 git 호스트로 `git push --mirror` 복원 가능한 DR 체계 + 백업물 토큰 노출 제거. GitLab 자체 장애용 전체 백업과 **2단 백업 전략**으로 분리.

#### 6-2. GitLab 18+ 복원 후 텅 빈 레지스트리 — metadata DB 별도 복원

- **상황** GitLab 본체 backup tar만으로 복원하면 registry filesystem은 살아나지만 이미지 pull이 불가.
- **원인** GitLab 18+는 registry의 manifests/tags가 **별도 metadata DB**에 있는데 일반 backup에 미포함 → 파일은 있어도 메타데이터가 빈 상태.
- **해결** 본체 `gitlab-backup restore`(~12분)에 더해 registry filesystem 별도 해제·ownership 정정 + 원본의 `registry` DB를 `pg_dump`→신규 PostgreSQL에 restore. ConfigMap의 `gitlab_https/protocol/port`를 외부 클라이언트 기준(https/443)으로 정정해 URL redirect 사고도 해소.
- **결과** 7 repos / 63 tags / 56 manifests / 572 blobs를 원본과 동일하게 신규 클러스터에서 복원.

#### 6-3. PostgreSQL Pod 기동 실패 — ext4 `lost+found` + PVC 권한

- **원인** ① ext4 포맷 시 생기는 `lost+found`가 initdb "빈 디렉토리" 조건과 충돌, ② PVC 마운트가 root 소유라 postgres user write 권한 부재.
- **해결** `PGDATA`를 서브디렉토리(`/var/lib/postgresql/data/pgdata`)로 옮겨 `lost+found` 회피, `securityContext.fsGroup: 999`로 권한 부여. 전용 노드에 데이터볼륨을 UUID 기반 fstab으로 마운트, `postgres:17-alpine` 운영(KT mirror에 17.6 부재 대응).
- **결과** `pg_dumpall` import로 32 테이블 정상 기동.

#### 6-4. 배포·스토리지 운영 정책 (묶음)

- **캐싱된 과거 이미지 배포** — 노드에 캐싱된 과거 이미지가 쓰여 신버전 미반영. `imagePullPolicy: Always`로 매 배포 시 최신 이미지 강제(백엔드·프론트에도 동일 적용).
- **PVC 삭제 시 데이터 소실** — Jenkins NFS SC의 Reclaim Policy가 `Delete`라 PVC 삭제 시 데이터 초기화. 운영 `nfs-retain-prod`(Retain) / 개발 `nfs-delete-dev`(Delete)로 분리, Jenkins 재설치 시 Retain 적용. (이 분리가 이후 미러 PV·DR 체계의 기반)

### 성과

7노드 규모 클러스터·CI/CD·데이터·백업 전체를 코드로 재현 가능하게 GitOps화하고, 토큰 비노출 미러 백업으로 임의 git 호스트 복원이 가능한 DR 체계를 구축했다.
